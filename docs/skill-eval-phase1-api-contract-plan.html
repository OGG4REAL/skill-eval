实施结论

已按方案 B 落地：在现有 BenchmarkStore 和 SkillComparator 上补 summary/contract 序列化能力，server/app.py 只加薄路由，frontend/src/lib/api.ts 只加类型化调用。

结果：新增 6 个 /evaluation/* API、前端 typed API client 和 API smoke test。下一步是让 Evaluation 页面消费这些 contract；当前仍不引入 DB、queue 或异步 job。

成功标准

  • 新增 6 个 /evaluation/* API，并返回稳定 contract。
  • overview、benchmark、comparison 都有 UI 可直接消费的 summary。
  • 前端 API 封装只消费 summary/detail contract，不读 benchmark 原始 JSON 文件。
  • 现有 evaluation 测试通过。

非目标

  • 不引入数据库。
  • 不加后台队列或 worker。
  • 不重写 BenchmarkRunner。
  • 不在 Phase 1 重做前端页面骨架。

已定风险处理

  • POST /evaluation/benchmarks/run 允许同步阻塞执行；Phase 1 不加队列和后台状态轮询。
  • GET /evaluation/comparisons 默认使用全量 compare_all，支持 benchmark_id 做单次 benchmark 对比。
  • GET /evaluation/benchmarks/{id} 不透传完整 raw cases，只暴露 matrix、failed case summary、run refs。

方案 A：路由内拼装

不采用

优点

  • 改动最少。
  • 不新增模块。
  • 短期能很快打通 API。

缺点

  • server/app.py 会继续膨胀。
  • summary 逻辑难单测。
  • 后续 Phase 3/4 容易复制前端拼装逻辑。

方案 B：Store/Comparator contract helper

已采纳

优点

  • 不引入新架构。
  • summary 贴近数据来源。
  • 后端路由保持薄。
  • 能直接扩展现有 evaluation 单测。

约束

  • core 模块里允许出现少量 API contract 字段。
  • 必须避免把 raw JSON 全部透出。

方案 C：新增 API service 层

不采用

优点

  • contract 和 core 计算分离。
  • 后续 schema version 更好放置。
  • 更适合 Phase 7 hardening。

缺点

  • 新增文件和概念。
  • Phase 1 规模偏重。
  • 容易提前平台化。

已定 API Contract

Endpoint 用途 核心 response 已定处理
GET /evaluation/overview Overview 第一屏，不让前端临时拼指标。 summary、latest_benchmarks、comparison_summary、warnings。 缺 benchmark 时返回 200 空态，summary 内计数为 0，warnings 可提示暂无数据。
GET /evaluation/benchmarks Benchmark 列表。 每项包含 benchmark_id、started_at、scope、summary、comparison_summary。 不返回 _path 给前端，避免前端绑定本地文件路径。
GET /evaluation/benchmarks/{id} Benchmark detail 和 Phase 3 matrix 数据入口。 summary、matrix、failed_cases、run_refs、comparison。 只暴露筛过的 case summary，不透传完整 case；错误信息保留必要摘要和 run reference。
GET /evaluation/skills/{skill}/summary Skill 维度汇总。 summary、tasks、recent_benchmarks、warnings。 未知 skill 返回 200 空态，便于 UI 搜索和 filter。
GET /evaluation/comparisons Comparison 页面数据。 summary、by_skill、by_task、source、generated_at。 默认 source=all_benchmarks；支持 ?benchmark_id=... 做单次 benchmark 对比。
POST /evaluation/benchmarks/run 同步触发 runner，保持无队列。 请求完成后返回新 benchmark 的 summary 和 benchmark_id。 接受同步长耗时请求；请求完成后直接返回 benchmark contract。Phase 1 不做异步 job。
已定 TypeScript response 形状 
interface EvaluationOverviewResponse {
  summary: {
    benchmark_count: number;
    task_count: number;
    skill_count: number;
    latest_benchmark_id: string | null;
    latest_started_at: string | null;
    cases_total: number;
    cases_failed: number;
    positive_comparisons: number;
    negative_comparisons: number;
  };
  latest_benchmarks: EvaluationBenchmarkListItem[];
  comparison_summary: EvaluationComparisonSummary;
  warnings: string[];
}

interface EvaluationBenchmarkListItem {
  benchmark_id: string;
  started_at: string | null;
  finished_at: string | null;
  scope: {
    task_id: string | null;
    group: string | null;
    all: boolean;
    variants: string[];
    trials: number;
  };
  summary: {
    cases_total: number;
    cases_succeeded: number;
    cases_failed: number;
    pass_rate: number | null;
    task_count: number;
    variant_count: number;
  };
  comparison_summary?: EvaluationComparisonSummary;
}

interface EvaluationComparisonSummary {
  source: string;
  generated_at: string | null;
  baseline_variant: string;
  target_variant: string;
  tasks_compared: number;
  skills_compared: number;
  positive_tasks: number;
  negative_tasks: number;
  neutral_tasks: number;
  avg_result_score_uplift: number | null;
  avg_normalized_gain: number | null;
}

已定决策清单

Run endpoint。 允许同步阻塞执行；不引入 queue、job id、后台 worker。
Comparison 默认范围。 默认全量 compare_all；支持 benchmark_id 查询单次 benchmark 对比。
Benchmark detail 信息边界。 暴露 matrix、failed case summary、run refs；不透传完整 raw JSON。
未知 skill 返回语义。 返回 200 空态，而不是 404。
API contract 测试。 在不扩大范围的前提下补小型 smoke test；仍以现有 evaluation 测试通过为主验收。
summary 字段命名。 按本文 TypeScript contract 固定，后续 Phase 2/3 直接依赖。

明确不做的事

  • 不把 evaluations/benchmarks/runs/*.json 文件路径返回给前端。
  • 不让 frontend 自己推导 task count、pass rate、uplift。
  • 不为了未来 regression 先做通用查询语言。
  • 不把 BenchmarkRunner 改成异步任务系统。
  • 不在 Phase 1 改 EvaluationPanel 的 UI 形态。
